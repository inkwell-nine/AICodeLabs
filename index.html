    # How to work with me on this codelab

I'm a finance professional doing one of the AI Codelabs. I am Excel-fluent and a smart investor or operator, but I'm not a developer. I haven't written Python. I've barely lived in a terminal. I've used CLIs only by following someone else's instructions.

I'm doing this codelab to get hands-on with modern LLM tools and to build a real point of view on the investing and operating implications of AI. The goal isn't for me to leave as a developer — it's for me to leave with a feel for the *shape* of these tools and where they go to work in our business.

Here's how to work with me.

## How to talk to me

- **Lead with the Excel analogy, then the real term.** Whenever you introduce a new concept — a file, a command, a programming idea, a piece of architecture — describe it first as "this is the Excel equivalent of X" and *then* tell me what the technical name for it is. There's a cheat sheet at the bottom of this file. Extend it as we go.
- **Explain why before what.** Before you install something, write a script, or take an action, give me one sentence on *why* this is the right move. Then do it.
- **Plain English by default.** No jargon without unpacking. If you have to use a term I might not know, define it in the same sentence ("a tokenizer — the thing that counts how a model charges you per word").
- **Narrate when you act.** When you run a command, write a file, or call a tool, tell me in one sentence what you just did. I want to feel the pace of what's happening, not have it dump silently.
- **Don't tell me things are "simple," "easy," or "trivial."** Let me decide. Also avoid "just" in front of a technical step.
- **Push back when I'm asking the wrong question.** If I ask you to do X but the right move is Y, say so. I'm not technical, so I'll ask for the wrong thing sometimes. I'd rather you correct me than humor me.
- **Don't apologize for complexity.** If something is genuinely hard, name what's hard about it and proceed. No "sorry, this gets a little technical."

## What good looks like

**Less good:** *"I'll just pip install fireworks-ai and add it to your `.env`."*

**Better:** *"I'm about to install the Fireworks Python library — think of it as an Excel add-in that gives us a custom function for talking to Fireworks. Small, quick, no big deal. I'll then save your API key to a `.env` file, which is basically an external linked workbook where you keep your passwords so they never appear in the main file. Going ahead."*

**Less good:** *"That's a tokenization issue."*

**Better:** *"That's a tokenization issue — the model splits your text into chunks before it reads it, and different models count chunks differently. Same sentence might be 30 chunks to one model and 22 to another, which changes the price. Want me to show you the count?"*

## Default mode: Excel analogies (when they hold up)

When I look confused, when something feels abstract, when I ask "what is that?" — your default is to translate the concept into Excel terms *if* there's a clean mapping. Below is a cheat sheet of analogies that earn their keep. Use them. If a concept I hit isn't on the list and you can't think of a clean Excel analog, **don't force one** — explain it in plain English instead. A stretched analogy is worse than no analogy.

| Programming idea | Excel equivalent |
|---|---|
| **File / folder** | A workbook / a folder of workbooks. |
| **`.env` file** | An external linked workbook that holds my passwords and API keys. The main workbook pulls them in via external references so they never appear in the visible sheet. |
| **Variable** | A named cell. |
| **Function** | A custom function defined in the Name Manager — inputs go in, an output comes out. |
| **Loop (for / while)** | A formula filled down across many rows. |
| **If-statement / conditional** | An `IF()` formula. |
| **List / array** | A column or a horizontal range. |
| **Dictionary / object** | A two-column lookup table — keys on the left, values on the right. `VLOOKUP` is how you read it. |
| **Script (Python file)** | A macro module — a sequence of steps you can re-run. |
| **Library / package** | An add-in like Solver or Analysis ToolPak — code someone else wrote that you can call. |
| **Terminal** | The window where you type commands directly to your machine, the way you type formulas into the formula bar. |
| **CLI (command-line tool)** | A formula you type into the terminal that runs a specific tool. |
| **API call** | A "Get Data > From Web" external data connection — you go fetch something from a server. |
| **API key / auth token** | The credentials saved in the linked credentials workbook (see `.env` above). |

For LLM-specific ideas (tokens, tokenizers, embeddings, RAG, harnesses, agents, etc.) — these are concepts the codelab is *designed to teach*, and the right move is to explain them on their own terms, not to bolt an Excel analogy onto them. I want to learn what these things actually are.

## When to pause and check with me

Before any of these, stop and ask:

- Installing anything that takes more than ~30 seconds or more than ~100 MB.
- Modifying anything outside this folder.
- Spending API budget likely to exceed a few dollars in a single call.
- Pushing anything to a remote server I don't control (a hosted GitHub repo, a deployed endpoint, etc.).
- Any irreversible action — deleting files, dropping a database, sending an email.

## What I'll do on my end

- I'll paste errors and unfamiliar terms back to you and ask you to explain in plain English.
- I'll let you drive when there's a CLI or an API to use — I won't click through UIs by hand.
- I'll tell you when I'm confused so you can re-explain in Excel terms.
- I'll be patient. I'll know nothing on the first try. That's the exercise.

## A reminder for both of us

This is a learning exercise, not a production project. Pick the path that teaches me the most about the *shape* of the thing, not the path that produces the cleanest artifact. If two approaches both work but one is more revealing, use the more revealing one.

    Why --dangerously-skip-permissions --effort max

    You’ll see this command in every codelab’s “Before you start” section. The reasoning is the same every time, so we’ve put it here once.

    Why these flags

    --dangerously-skip-permissions turns off the prompt that asks before every action Claude takes — running a command, writing a file, installing something. The name sounds alarming because in some contexts it would be. Here it isn’t: you’re working in a fresh folder dedicated to this exercise, and the worst case is that you delete the folder and start over. The constant approval prompts would derail the flow and rob you of the let-Claude-drive feeling that is the whole point of these codelabs.

    --effort max tells Claude to think as hard as it can on every step. During a learning exercise you want the strongest version of the model, not the cost-saving one.

    The mindset, in three sentences

    • Tool-first, not hand-first. When a platform has a CLI, install it and hand it to Claude. When it has only an API, generate a key and let Claude write the calls. You almost never click through a UI.
    • Stuck is the lesson. When something breaks or doesn’t make sense, paste it back into Claude and ask. The way out of friction is always the agent, not Google.
    • You provide judgment; Claude provides typing. Decisions, observations, taste, and ownership are yours. Recording, scripting, formatting, timing — those go to Claude.

    Chapter 1

    Foundation

    Chapter 1 is the foundation. The goal is to come out knowing how to download an open-source model, how to get it running on your laptop, and which model sizes fit which kinds of hardware. Everything downstream assumes you’ve handled at least one model end-to-end.

    01Open-source models & local hosting

    Pull a small open-source model from Hugging Face onto your laptop and get it answering prompts. The exercise is mostly about two practical things: how to download a model via the CLI rather than clicking around the website, and which model sizes are realistic for which kinds of machines — the memory math that decides whether 3B is your ceiling or whether you can run a 7B comfortably. A comparison against Kimi 2.5 on a Fireworks endpoint at the end gives you a feel for what local can and can’t do.

    Firm lens Running models locally is a real option for sensitive deal materials, but the memory ceiling on a laptop is much lower than the cloud — which means you’re trading model capability for data control. Knowing roughly which sizes fit where lets you make that trade-off honestly when it comes up. Also useful when evaluating portcos or vendors that claim “edge-deployable” or “runs on your hardware.”

    Chapter 2

    The access layer

    Most production AI systems aren’t pointed at one provider. They go through an intermediary that fans out to many. Once you’ve felt the model-versus-hosting distinction, the next move is reaching every model with a single key.

    02Model routing

    Set up OpenRouter — one API key, every major model — and build a router that sends each prompt to the cheapest model that meets a quality bar. We do this codelab early because the next two depend on having OpenRouter set up.

    Firm lens Routing is the pattern most production AI systems use, including ones a portco might be building. When you see “we use Anthropic” (or any single provider) in a vendor’s architecture diagram, it usually should mean “via a router we could swap in 30 minutes if pricing or quality shifted.” Useful when reading AI-platform diligence and when thinking about the cost line on internal tools.

    03Performance & cost benchmark

    Run the same ten prompts through ten OpenRouter models. Measure cost per token, latency, tokenizer differences, and spot-check quality. Now that you can reach every model, the next question is which one to use for which kind of work.

    Firm lens Cost across models can vary by 100× for the same prompt. Without having seen the spread once, you don’t have a baseline to size an AI budget for a deal team, evaluate a portco’s AI cost claims, or sanity-check what an AI-native investment thesis is actually paying for inference.

    04Harness–model coupling

    Swap the model behind Cursor and behind Claude Code on the same coding task. Watch how the same harness behaves differently with different models behind it. The point is that the harness and the model are two different choices.

    Firm lens Two AI products that feel meaningfully different are usually differing on one of three things: the model, the harness, or the prompt scaffolding around both. Being able to tell them apart is useful when comparing vendors and when explaining why a tool you tried six months ago feels different now.

    Chapter 3

    Inside the agent

    Up to this point you’ve worked with models from the outside — using harnesses, swapping models behind them, comparing them. The next three codelabs go inside the harness itself.

    05Harness from scratch

    Build a small ReAct-style harness with two tools (web_search and get_local_time). Claude writes the code; you read it. The point is to see that a harness is roughly a while loop with some glue around it. Codelab #4 introduced the abstraction; this one makes it concrete.

    Firm lens Useful for diligence on AI-native companies. When a vendor’s pitch leans on “our secret sauce,” you’ll have a sharper read on which parts are plumbing — agent loops, tool-calling — and which parts are actually differentiated.

    06MCP server build

    Wrap the National Weather Service API in an MCP server and connect it to Claude. Watch the same prompt go from a vague guess to a real forecast. Now that you’ve built a harness with tools by hand, MCP is the standard way other harnesses pick up new tools.

    Firm lens MCP is how AI tools get connected to internal systems: PitchBook, Salesforce, SharePoint, deal-room files. Most conversations about “can we connect Claude to our X?” come down to whether X already has an MCP server and whether someone could build one in an afternoon. Doing it once makes those conversations concrete.

    07Memory & retrieval

    Implement two or three retrieval patterns on the same corpus — vector RAG, markdown-wiki, optionally graph RAG. Compare quality, latency, and setup cost. Tools let an agent do things; retrieval lets it look things up. Most useful firm workflows need both.

    Firm lens Most “AI for our documents” projects succeed or fail on retrieval, not on the model. The right pattern depends on the shape and size of the corpus, and getting it wrong means the model never sees the right context. Worth understanding before you greenlight an internal RAG project or evaluate one a portco is proposing.

    Chapter 4

    Packaging and platforms

    Codelabs 1 through 7 cover the same building blocks: model, harness, tools, memory. The next two are about packaging your work and running it elsewhere — different platforms, different file formats, mostly the same underlying abstractions.

    08Plugins & portable agents

    Build a small Claude plugin with one agent and one skill. Watch the agent run in its own context window, separate from your main session. Then translate the agent definition into Codex (OpenAI’s CLI) and run the same task there. The lock-in between harnesses for skills and agents turns out to be small — the wrapper file is vendor-specific, but the agent itself moves with a short translation.

    Firm lens When you build something useful in Claude Code for a recurring workflow at the firm, this is the path to packaging it so a teammate can use it. More importantly: when a portco or vendor pitches that their product is “built on Anthropic” (or OpenAI, or Gemini), this codelab tells you what that lock-in is actually worth. Usually a lot less than the vendor implies.

    09Hermes Agent deep-dive

    Spin up Hermes, dispatch a long-running task, walk away from your computer. Compare its trajectory to Claude Code on the same task. Codelab #8 packaged an agent so it could move across harnesses; this one explores a different dimension — interactive sessions you supervise versus async tasks you check back on later.

    Firm lens The firm has access to multiple agent platforms, and the question of which to use for which kind of work is a real one. Doing this codelab gives you a frame for that conversation, based on hands-on comparison rather than vendor decks.

    Chapter 5

    Compose and measure

    The last two codelabs do two different things. The first composes multiple agents into one workflow. The second teaches you how to decide whether a workflow is good enough to use.

    10Multi-agent SDLC pipeline

    Five Claude Code subagents — Planner, Builder, PR-opener, Reviewer, Merger — running a PRD-to-PR pipeline on a small real task. Cost meter open. You leave the computer for 20 minutes and come back to whatever they produced. This puts the earlier codelabs together, with a real cost attached.

    Firm lens Internal tools at $5–50 per build, running unsupervised. The codelab is mostly about feeling the cost-to-output ratio for yourself, which is hard to grasp from articles. Useful when evaluating engineering-heavy portcos and when thinking about what kinds of internal tools the firm itself could build this way.

    11Evals as a workflow

    Build a small eval suite for a real workflow — a Sources & Uses table build is the default — scored against a rubric. Run the same suite across three models. Rank them. The point is that without an eval, every model-choice or prompt-tweak conversation is a vibe; with an eval, you have a number.

    Firm lens Most AI workflows worth rolling out at the firm — internal tools, portco diligence, automated analyses — should have an eval before they go past one user. The harness for evals is small; the discipline of writing the rubric is the hard part. Worth doing once before you scale any AI workflow to a team.

    Time
    60–90 minutes
    Difficulty
    Beginner
    Prerequisites

    Why this matters

    Almost everyone here has used a language model through a chat box — Claude.ai, ChatGPT, Gemini. Those are closed models running on someone else’s servers, billed by the token. There is a parallel universe of open-source models that you can download and run on your own laptop, or on a cheap hosted endpoint, for a fraction of the cost.

    You don’t need to become an open-source-model expert. But the moment you’ve done this once, your mental model of “how AI gets bought and built” gets meaningfully sharper. You’ll know, viscerally, that “the model” and “where the model runs” are two separate choices — which matters when you’re evaluating a vendor, thinking about cost lines, or worrying about whether sensitive data should leave the building.

    What you’ll do

    • Pull a small open-source language model from Hugging Face onto your laptop and get it answering prompts.
    • Call a much larger frontier model (Kimi K2 / Kimi 2.5) on a hosted Fireworks endpoint.
    • Send the same 5 prompts to both and compare latency, cost, and quality.
    • Walk away with a real point of view on when small-and-local is enough and when you reach for a frontier model.

    What you’ll actually learn

    The exercise is the surface. The real takeaways:

    • How to download and run an open-source model on your laptop. The mechanics — CLI, Hugging Face, choosing a runtime, getting a first response back. This is the foundation everything downstream relies on.
    • How to estimate whether a model fits on a given machine. Parameter count × bytes per parameter ≈ memory needed. A 7B model at 4-bit quantization is ~3.5 GB; a 70B is ~35 GB; a 405B is firmly past laptop territory. This math comes up every time you’re evaluating whether something is doable on the hardware in front of you.
    • Hand the agent the CLI. When you meet any new platform, the move is to install its command-line tool and let Claude drive it. You don’t click through websites.
    • You can ask Claude to explain itself. Every term, every error, every weird output — paste it back and ask. This is the loop.

    Milestone 2 · Run a small model on your laptop

    ~20 min

    Goal

    An open-source language model is running on your machine and answering prompts. It can be slow. It can be weird. It just has to work.

    Pick from this list

    These are small, well-known, instruction-tuned models that actually run on a normal laptop. Don’t pick yourself — share the list with Claude and let it choose based on your machine.

    • Llama 3.2 3B Instruct (Meta) — easy default. Solid quality, runs on almost anything.
    • Qwen 2.5 7B Instruct (Alibaba) — punches above its weight if you have ≥16 GB RAM.
    • Gemma 2 2B (Google) — very small and fast. Great on older or RAM-constrained machines.
    • Phi-3.5 Mini Instruct (Microsoft) — small, instruction-tuned, surprisingly capable for its size.
    • Llama 3.2 1B Instruct — fallback if your machine is really tight on memory.

    The memory math (and what you’re not running)

    Before you download anything, it’s worth knowing why the list above stops at 7B. The math is roughly: parameter count × bytes per parameter ≈ memory needed just to load the weights.

    • At FP16 (the default precision): ~2 bytes per parameter. A 7B model loads in roughly 14 GB of RAM.
    • At 4-bit quantization: ~0.5 bytes per parameter. The same 7B model now fits in ~3.5 GB.
    • Add a few GB on top for activations, the KV cache (which grows with context length), and the OS.

    For scale, here’s where some recently-released state-of-the-art models sit:

    • Llama 3.3 70B (Meta) — ~140 GB at FP16, ~35 GB quantized. A workstation, not a laptop.
    • Mistral Large 2 — 123B parameters. Similar memory profile to Llama 70B.
    • Llama 3.1 405B (Meta) — the largest dense open model widely available; ~800 GB at FP16, ~200 GB quantized. Datacenter territory.
    • DeepSeek V3 / R1 — 671B total parameters, ~37B active per token (mixture-of-experts). All weights still need to live in memory, even though only ~37B fire on any given token.
    • Kimi K2 / 2.5 (Moonshot) — roughly 1T total parameters, ~32B active. The model you’ll call on Fireworks in Milestone 3.
    • Frontier closed models (Claude Opus 4.6, GPT-5, Gemini 2.5) — exact sizes aren’t disclosed, but the consensus is trillions of parameters, MoE. Datacenter territory regardless.

    The model you’re about to run on your laptop is roughly 100× to 1,000× smaller than the frontier. That’s worth feeling. It’s also why this codelab’s comparison at the end isn’t fair in absolute terms — it’s showing you the shape of a trade-off, not the limit of what local models can do.

    Nudges

    • Hand the list above to Claude and let it pick. Ask it to explain the trade-off it just made (size vs. quality vs. how much RAM your laptop has).
    • Before you actually download, ask Claude to do the memory math out loud for the model it picked. You should be able to roughly check the calculation yourself by the end.
    • The first run will probably feel slow. Ask Claude why. If “quantization” comes up, ask Claude to explain that in one paragraph and whether you should use a quantized version (you probably should).
    • If Claude wants to install a heavyweight runtime, ask it to pick the lightest tool that gets the job done — llama.cpp or ollama are both fine; you don’t need anything bigger.

    Suggested prompts

    Paste into Claude

    I want to run a small open-source language model on my laptop. Here’s the shortlist of models I’m choosing from — Llama 3.2 3B, Qwen 2.5 7B, Gemma 2 2B, Phi-3.5 Mini, Llama 3.2 1B. Look at my machine, pick the one that’ll actually run well here, explain why you picked it, and get it set up so I can send it a prompt. Use the lightest runtime that works.

    Paste into Claude

    Before we download anything, walk me through the memory math for the model you just picked. How many parameters? At the quantization you’re recommending, roughly how much RAM does that take just to load the weights? Then sanity-check: does my machine have enough headroom for that, plus context and the OS? If we’re cutting it close, switch to a smaller model and tell me what I’m giving up.

    Paste into Claude

    The first response was really slow. Explain in one paragraph why local models are slower than the cloud ones I’m used to, and tell me whether a quantized version would help here. If yes, swap it out and re-run.

    Try this

    After you get a response back from your local model, ask it a question only a person at our firm would care about — something about a sector, a deal pattern, or a workflow. Notice how its answer compares to what Claude.ai would give you. Sit with that gap for a minute. That gap is what this codelab is teaching you to feel.

    Milestone 3 · Call Kimi 2.5 on Fireworks

    ~15 min

    Goal

    You can call Kimi K2 / Kimi 2.5 — a frontier-scale open-source model that almost certainly won’t run on your laptop — via Fireworks, using nothing but an API key.

    This is the contrast that makes the comparison interesting. The model you just ran locally is small enough to fit on a laptop. Kimi 2.5 is hundreds of times larger. Both are open-source. Both are accessible to you in the next 15 minutes.

    Nudges

    • You don’t need to install the Fireworks CLI. It’s a heavier tool than you need. All you have to do is create an API key on the Fireworks dashboard and store it somewhere Claude can pick it up.
    • Don’t paste the key into a chat message. Ask Claude where it should live (a .env file or a shell environment variable) and let Claude write the code that reads it.
    • If Fireworks lists multiple Kimi variants, pick the most recent one. Ask Claude to confirm the exact model identifier before it writes the call.

    Suggested prompts

    Paste into Claude

    I’m about to create a Fireworks API key. Tell me exactly where to click on the Fireworks dashboard, and then set things up on my machine so that you can read the key securely without me ever pasting it into our chat.

    Paste into Claude

    My Fireworks key is stored. Write a small script that sends a prompt to Kimi 2.5 on Fireworks. Confirm the exact model identifier before you call it. Start with “Hello, who are you?” so I can confirm it works.

    This milestone is where most people start to feel the rhythm. You’re repeating the move from Milestone 1 — give Claude what it needs (this time a key instead of a CLI) and let it drive. If it feels easy, that’s not because the work is trivial. It’s because you’ve already learned the move.

    ✓ You can send “Hello, who are you?” to Kimi 2.5 on Fireworks and get a response back. Notice how much faster it is than the local model.

    Milestone 4 · The comparison

    ~20 min

    Goal

    Milestone 3 · Write the router

    ~20 min

    Goal

    A function — call it route(prompt) — that takes a prompt and returns the answer from the cheapest model that meets a quality bar. It should print, for every call, which tier it picked and why.

    The routing logic, in plain English

    There are two reasonable strategies. Both are worth knowing.

    1. Classify first. Use a cheap model to look at the prompt and decide what kind of question it is (simple lookup / mid-complexity / hard reasoning). Then send it to the matching tier. One extra call, but predictable.
    2. Cheap first, escalate. Send every prompt to tiny first. Have a cheap model judge whether the answer looks confident and complete. If not, escalate to mid. If still not, escalate to frontier. Slower in the worst case, but cheaper on average.

    Pick one for now. Classify-first is easier to reason about for a first build. Cheap-first / escalate is closer to how real systems work but has more moving parts. Either one teaches the lesson.

    Nudges

    • Don’t hand-write the classifier. Ask Claude to draft the classifier prompt — “given a user prompt, return one of simple / medium / hard and a one-line reason.” Use your tiny model for the classifier itself — the routing decision should not cost more than the answer.
    • Have the router print its work. Every call should produce one line of output like [router] simple → tiny ($0.00012) — “factual lookup, no reasoning needed”. The whole point of this codelab is that you watch the decisions happen.
    • Don’t over-engineer. Three tiers, three rules, ten lines of code. If Claude wants to write fifty lines, tell it to cut.

    Suggested prompts

    Paste into Claude

    Paste into Claude

    Why this works

    You just built a tiny version of what every serious production AI system does. The classifier is itself an LLM call — a model judging which model the “real” call should go to. That feels recursive and weird the first time you see it. It’s also exactly right. Cheap models are great at routing decisions even when they’d be lousy at the actual reasoning.

    Harness–model coupling

    Time
    60–90 minutes
    Difficulty
    Beginner
    Prerequisites
    Claude Code installed, comfortable opening a terminal
    You will need
    Cursor installed (free tier is fine), an OpenRouter or Fireworks account for API access

    Why this matters

    When most people say “the AI,” they’re collapsing two completely different things into one word. There is the model — Claude, GPT, Llama, Kimi — the thing that produces text. And then there is the harness — Claude Code, Cursor, OpenCode, Cline — the program wrapped around the model that gives it tools, parses what it says, and turns its output into real actions like editing a file or running a command. These are independent choices. You can pair almost any model with almost any harness.

    That mental model is the difference between “ChatGPT is bad at this” and “Cursor with GPT-5.5 was bad at this, but Cursor with Opus 4.7 nailed it on the first try.” When a portfolio company tells you they evaluated AI coding tools and it didn’t work, your follow-up is now sharper: which harness, which model, and did anyone try a different pairing? Same goes for vendor pitches. Same goes for your own day-to-day. The defaults you get out of the box are choices someone else made for you — usually fine, sometimes wrong for the task in front of you.

    What you’ll do

    • Give the same small, finance-flavored coding task to three different models inside Cursor and watch the trajectories diverge in real time.
    • Repeat the experiment from Claude Code with a model override, so you feel that the harness and the model are genuinely separate dials.
    • Compare the runs on a few axes that matter: steps taken, tokens burned, time elapsed, and whether the result was actually usable.
    • Walk away able to explain — to a colleague or a vendor — the difference between “the model” and “the harness” without hand-waving.

    What you’ll actually learn

    The exercise is the surface. The real takeaways:

    • Model and harness are two dials, not one. The same model behaves differently in different harnesses. The same harness behaves differently with different models. You get to choose both.
    • Defaults are choices you can override. Every tool ships with a default model. That default is a reasonable guess by the vendor — not a law of physics. Knowing how to swap it is a small skill with big leverage.
    • Trajectory matters as much as the answer. Two agents can both “solve” a task and burn wildly different amounts of time, tokens, and code. The one that took eleven steps to write thirty lines is not the same as the one that took three steps to write the same thirty lines, even if the final file is identical.

    1. Claude Code installed and working.
    2. Cursor installed. The free tier is enough for this exercise; you don’t need Pro.

The constant approval prompts would derail the flow and rob you of the let-Claude-drive feeling that is the whole point of these codelabs.

    --effort max tells Claude to think as hard as it can on every step. During a learning exercise you want the strongest version of the model, not the cost-saving one.

    Time
    60–90 minutes
    Difficulty

    Why this matters

    Every conversation about “let’s use AI on our deal files” ends in the same place: “but how does it find anything?” Someone at the table says the word “RAG” with confidence and the rest of the table nods. Two weeks later a vendor quotes you six figures for a vector database and an embeddings pipeline, and nobody around that table can tell you whether you actually needed any of it.

    You don’t need to become a retrieval engineer. You do need to know that “RAG” is not one thing — it’s a design space, with at least three reasonable points in it, and the cheapest one often beats the fanciest one for the corpus sizes we actually work with. After this exercise you’ll have run all three on the same data and seen the difference with your own eyes.

    What you’ll do

    • Have Claude generate a small, realistic synthetic portfolio — 20 markdown files, one per company, covering different sectors, vintages, and financial profiles.

    • “RAG” is a design space, not a product. The default “embed everything and do cosine similarity” is one point in that space. There are others. Knowing they exist is the difference between buying the right thing and the expensive thing.
    • Simpler patterns often beat fancier ones at our scale. A well-organized folder of markdown files plus an agent with read access can outperform a vector database for small corpora, with zero infrastructure to maintain.
    • “How does the agent find things” is upstream of “which model do we use.” Get retrieval wrong and the smartest model in the world can’t help you. Most AI failures at firms like ours are retrieval failures, dressed up as model failures.

    Paste into Claude

    Milestone 3 · The markdown-wiki pattern

    ~15 min

    Goal

    The same five questions, answered by Claude Code itself, with no embeddings, no vector store, and no Python script — just file-system tools pointed at the corpus/ folder. This is Andrej Karpathy’s “markdown wiki” pattern, and it’s deceptively powerful.

    Nudges

    • You’re already in Claude Code. It already has read access to the folder. There is nothing to install. Ask Claude to do a little prep work first: add a top-level INDEX.md that lists every company, its sector, and its vintage, so Claude has a fast way to triage which files to open.
    • For each question, ask Claude to show its work — which files it opened, in what order, and why. This makes the comparison with vector RAG honest.

    Suggested prompts

    Paste into Claude

    I want to try a different retrieval pattern. Instead of embeddings and a vector store, I want you — Claude Code, with your file tools — to be the retrieval system.     Paste into Claude

    Why this works

    You’re using the agent itself as the retrieval engine. Its “index” is your folder structure plus the INDEX.md you just had it write. It doesn’t need embeddings because it can read. For small, well-organized corpora — which is what almost every internal-knowledge use case at our firm actually is — this approach is shockingly hard to beat. It also has a property vector RAG doesn’t: it can reason about which files to open, instead of just matching tokens.

    Nudges

    Milestone 3 · Add the skill

    ~15 min

    Goal

    A SKILL.md file that codifies the workflow around the agent — when to use it, how to call it, what to do with the output.

    What a skill is, briefly

    If an agent is “a worker with a specific job,” a skill is “a documented procedure that uses workers.” Skills live above agents. They describe when to invoke an agent (e.g., “use this skill any time the user asks for a quick screen on a company”), what input the agent expects, and how to format the output. The main Claude Code session picks up skills automatically and invokes them when their triggers fire.

    Nudges

    • Ask Claude to write the SKILL.md by reading the official skill-format documentation. The conventions are evolving — don’t guess.
    • The skill should reference the agent by name and describe a clear handoff: the user asks for a screen, the skill catches that, the agent does the work, the skill formats the answer.
    • Keep the trigger description specific. “When the user asks about a company” is too broad; “when the user explicitly asks for a quick screen, deal screen, or initial take on a private company” is right.

    Suggested prompts

    Paste into Claude

    Why this works

    Context isolation is the unsung feature of subagents. Your main chat stays clean; the agent burns its own tokens on its own scratch reasoning; you get back just the result. Without isolation, every agent call would inflate your main context, get more expensive on every subsequent turn, and degrade reasoning quality.     Milestone 2 · Configure the five-agent pipeline

    ~20 min

    Goal

    Five named subagents defined inside your Claude Code project, each with its own role description and the right tools. A fresh GitHub repository connected to the folder. A short orchestrator prompt that hands the PRD to the planner and tells the planner to hand off when ready.

    The roster

    You’re building a tiny SDLC (software development lifecycle) on your laptop. Each role does one job:

    • Planner. Reads PRD.md. Writes PLAN.md — the file structure, the implementation steps, the testing approach. Does not write any product code.
    • Builder. Reads PLAN.md. Writes the actual code on a feature branch. Runs tests locally if any exist.
    • PR-opener. Commits, pushes the branch, and opens a pull request via gh with a clear title and description that references the PRD.
    • Reviewer. Reads the diff on the PR and leaves substantive line-level comments. Plays the adversary — not a rubber stamp. Asks the Builder to revise if the diff doesn’t meet the PRD’s acceptance criteria.
    • Merger. Reads the review thread. If criteria are met and no blocking comments remain, merges the PR. If not, kicks it back to the Builder with a written reason.

    Nudges

    • You don’t define these by hand. Ask Claude Code to create them as subagents (use /agents or the equivalent command in your version). Each agent should get a system-prompt-style role description and a restricted set of tools (the Planner doesn’t need gh; the Reviewer doesn’t need to write files).
    • If a Claude Code skill applies to a role (e.g., a code-review skill, a PRD-writing skill), attach it. Skills sharpen the agent without you having to write a 400-line prompt.
    • The Reviewer is the most important agent to get right. If it’s too soft, the loop closes too fast and you ship slop. Push Claude to write a Reviewer prompt that explicitly tries to break the diff.
    • For the GitHub repo: ask Claude to create a fresh empty one using gh repo create, connect your folder to it, and push an initial commit. Don’t do this in a browser.

    If you got stuck

    The single most useful prompt in this entire codelab — keep it in your back pocket:

    Paste into Claude

    Here’s exactly what I saw on my screen: [paste]. Explain in plain English what’s going on, then tell me what to try next. If you can fix it yourself, do that. If we’re burning tokens in a loop, stop the run first.

    Common stuck points:

    • The agents are looping and the meter is climbing fast. Stop the run with Ctrl-C. Ask Claude to look at the trace and tell you which two agents are bouncing the work back and forth.     • The Reviewer rubber-stamped a bad PR. That’s a prompt problem, not a model problem. Strengthen the Reviewer’s role description — explicitly forbid “LGTM” with no substantive comments, and require it to list at least two things it considered rejecting. Then run again.
    • The Builder went way out of scope. Your PRD’s “out of scope” section was too short. Add specifics. “Do not add tests beyond the three listed.” “Do not introduce new dependencies.” Specificity beats hope.
    • The pipeline stalled and a subagent is waiting on me. Read what it’s asking. If it’s a clarifying question that should have been answered in the PRD, answer it now and then go back and update the PRD for next time. If it’s a permission/credential issue, fix that and resume.

    • Inspect the spread, look at where models disagreed with the judge, and write a one-paragraph point of view on which model you’d actually deploy for this workflow at the firm.

    Milestone 1 · Pick the workflow and draft the rubric

    ~20 min

    Goal

    You have a written rubric — 4 to 6 criteria, each scored 1 to 5 with explicit anchors at 1, 3, and 5 — saved as rubric.md in your folder. The rubric describes what “good” looks like for the workflow you picked.

    Pick from this list

    The default is Sources & Uses because the math is bounded and there’s an obvious right answer. If you’d rather pick a different bounded workflow, here are three that work just as well:

    • Spend real time on the rubric. This is the part most people skip and it’s the part that matters most. If your rubric is mush, your scores will be mush.
    • Each criterion needs an explicit anchor at 1, 3, and 5. “Accuracy: 5” is useless. “Accuracy: 5 = every line item correct and totals balance; 3 = totals balance but one line item misclassified; 1 = totals don’t balance” is a rubric.
    • Ask Claude to push back on your rubric. Vague criteria are the silent killer. A good prompt is “critique my rubric like a grumpy senior associate — where would two reasonable people disagree on the same output?”
    • Aim for 4–6 criteria. More than that and the judge gets noisy; fewer and you can’t tell models apart.

    Writing the rubric forces you to articulate what you actually want, which is something most people only do after they’re unhappy with a model’s output.     Milestone 2 · Build the test set

    ~15 min

    Goal

    A folder of 5–10 paired files: each input (a deal brief) sits next to an expert-graded reference output (the Sources & Uses you’d be proud to show). These reference outputs are your ground truth.

    Nudges

    • You don’t need real deal data. Three to five synthetic-but-plausible briefs — varying in size, capital structure complexity, and edge cases — will give you a meaningful signal. Ask Claude to generate them, then sanity-check each one.
    • The reference outputs are the part you must own. Don’t let Claude write them and then grade against them — that’s circular. Generate the inputs with Claude, then write or hand-correct the outputs yourself.
    • Mix easy and hard cases on purpose. One clean LBO, one with a refinanced bridge facility, one with rollover equity and earn-outs. Models look identical on easy cases; the spread shows up on the messy ones.
    • Keep the format consistent. Inputs all look the same; reference outputs all look the same. The judge will get confused if the structure wanders.

    Paste into Claude

    Milestone 5 · Inspect the spread and write the POV

    ~15 min

    Goal

    One paragraph — the kind you’d send to a colleague over Teams — on which model you’d deploy for this workflow at the firm, why, and where you saw it lose points.

    Nudges

    • The aggregate scores are the headline; the disagreements are the story. Look at any case where one model scored much lower than the others and read the judge’s justification. That’s where you learn what the model is actually bad at.
    • Also look at cases where all three models scored low. That’s probably a rubric problem or an input problem — not a model problem. A useful sanity-check.
    • Spot-check the judge. Pick two or three rows at random and grade them yourself against the rubric. If the judge consistently agrees with you, you can trust it. If it doesn’t, your rubric needs another pass.
    • The POV is not just “use Opus.” It’s “use Opus for this workflow because it wins on accuracy and totals-balance, but Sonnet is 80% as good for a fraction of the cost — so for high-volume routine cases I’d route to Sonnet and only escalate to Opus when totals don’t balance.” That sentence is the deliverable.

    Paste into Claude

    Look through results.csv and tell me: (1) which model wins overall, (2) which model wins on each individual criterion, (3) the three biggest disagreements between models — i.e., cases where one model scored very differently from the others — and what the judge said about each.

    Paste into Claude

    Paste into Claude

That’s the lesson: the rubric is a knob, and small changes to what you call “good” can flip which model looks best. The numbers are precise; precision is not the same as truth.

    Common stuck points:

    • My rubric feels mushy and I can’t tighten it. Ask Claude: “Give me three different model outputs for the same input — one clearly great, one mediocre, one bad. Score each one against my rubric and show your reasoning, so I can see where my anchors break down.”
    • The judge keeps returning prose instead of JSON. Ask Claude to add a structured-output constraint or a schema to the judge prompt, and to retry once with stricter instructions if parsing fails.
    • All three models scored almost identically and I can’t tell them apart. Either your inputs are too easy or your rubric is too coarse. Ask Claude: “Suggest three harder inputs that would force the models to differentiate, and tighten my rubric’s top anchors.”
    • One model wins on every criterion and it’s suspicious. Check that you’re not using the same model family as both the model-under-test and the judge. Ask Claude: “Could there be a self-preference bias here? Suggest a neutral judge and re-run.”
    • The eval cost is climbing faster than I expected. Ask Claude to print a per-model cost estimate before each run starts, and to cache judge results so you don’t re-score outputs you’ve already graded.
